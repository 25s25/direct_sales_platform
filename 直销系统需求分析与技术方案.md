# 直销系统通用框架 — 需求分析与技术方案

## 一、直销系统是什么？

**直销（Direct Selling）** 是一种绕过传统批发零售环节，由直销员（经销商/代理商）直接面向终端消费者进行产品推广和销售的商业模式。直销系统就是支撑这套业务运转的数字化平台。

**核心业务场景：**

```
           ┌──────────┐
           │  公司总部  │
           └────┬─────┘
                │ 发货 / 结算
    ┌───────────┼───────────┐
    ▼           ▼           ▼
┌───────┐  ┌───────┐  ┌───────┐
│ 经销商A │  │ 经销商B │  │ 经销商C │   ← 推荐关系
└───┬───┘  └───┬───┘  └───┬───┘
    ▼           ▼           ▼
 下线团队    下线团队    下线团队       ← 团队裂变
    ▼           ▼           ▼
 消费者      消费者      消费者        ← 终端零售
```

**区别于传统电商的关键点：**

| 维度     | 传统电商  | 直销系统                       |
| -------- | --------- | ------------------------------ |
| 销售主体 | 平台/商家 | 经销商个人                     |
| 组织关系 | 无        | 树状/网状推荐关系（上线-下线） |
| 收益模式 | 差价利润  | 零售利润 + 团队奖金（多层级）  |
| 结算对象 | 平台      | 多层经销商按制度分润           |

## 三、核心模块详细功能

### 3.1 会员管理（核心）

```
会员注册 → 实名认证 → 等级晋升 → 推荐裂变 → 团队管理
```

| 功能点   | 说明                                                                |
| -------- | ------------------------------------------------------------------- |
| 会员注册 | 手机号/邮箱注册，支持邀请码绑定推荐关系                             |
| 实名认证 | 身份证+银行卡四要素验证                                             |
| 会员等级 | 多级等级体系（如：普通→银卡→金卡→钻石→皇冠），支持自动升级/降级 |
| 推荐关系 | 记录推荐人、安置人，支持树形结构展示                                |
| 团队树   | 可视化展示下线团队结构，支持查看各级业绩                            |
| 安置模式 | 支持极差安置、双轨左右区安置、矩阵安置等                            |
| 会员权益 | 不同等级对应不同折扣、返利比例                                      |

### 3.2 奖金结算（直销系统灵魂）

**核心亮点：后台一键切换奖金制度。** 系统内置多种行业标准制度，管理员可在后台随时切换，无需修改代码或重启服务。

**内置奖金制度：**

| 制度类型         | 计算逻辑                               | 适用场景             |
| ---------------- | -------------------------------------- | -------------------- |
| **级差制** | 按等级设不同提成比例，上下级差额即奖金 | 传统直销，简单易懂   |
| **双轨制** | 分左右两区，按小区业绩计算对碰奖       | 快速裂变，团队协作   |
| **矩阵制** | 限制宽度和深度（如3×5），填满即溢出   | 强制互助，防止大象腿 |
| **太阳线** | 每个下级都是一条线，按线发奖金         | 个人能力驱动         |

**设计思路：** 采用「策略模式 + 配置驱动」架构，每种制度对应一个独立的奖金计算策略类，通过 `ds_bonus_plan` 表的 `type` 字段和 `calc_rule_json` 配置字段动态切换。切换制度后，新产生的订单按新制度计算，历史数据不受影响。

**奖金项设计：**

```
┌─────────────────────────────────────────────────┐
│              奖金结算体系                         │
├───────────┬───────────┬───────────┬─────────────┤
│ 零售利润   │ 推荐奖     │ 对碰奖     │ 层奖         │
│ (差价)    │ (推荐新人)  │ (双轨配对) │ (层级奖励)   │
├───────────┼───────────┼───────────┼─────────────┤
│ 领导奖     │ 分红奖     │ 重消奖     │ 见点奖       │
│ (代数提成) │ (全球分红)  │ (重复消费) │ (点位奖励)   │
└───────────┴───────────┴───────────┴─────────────┘
```

### 3.3 商品管理

- 商品分类（多级分类）
- SKU 管理（规格、价格、库存）
- 零售价 / 会员价 / 各级代理价
- 商品上下架、限购配置
- 库存预警、出入库记录

### 3.4 订单管理

- 购物车 → 下单 → 支付 → 发货 → 签收 → 分润
- 支持多种支付方式（微信/支付宝/余额/银行卡）
- 订单状态流转：待付款 → 已付款 → 已发货 → 已签收 → 已完成
- 售后：退货/换货/退款，奖金回退处理
- 订单业绩归属（按推荐关系计算提成）

### 3.5 财务管理

- 电子钱包（余额、冻结金额、奖金）
- 提现申请 → 审核 → 打款（支持批量）
- 收支明细、对账单
- 发票管理
- 税务代扣代缴

### 3.6 报表统计

- 经营总览（销售额、新增会员、活跃度）
- 团队业绩报表（按层级、按时间）
- 奖金发放报表
- 商品销售排行
- 数据大屏（实时经营数据可视化）

## 六、关键设计要点

### 6.1 奖金计算引擎（核心难点，策略模式实现制度切换）

奖金计算是直销系统的灵魂，需要做到：

- **可切换**：后台一键切换奖金制度，每种制度对应独立的策略类，继承统一接口 `BonusCalculator`
- **可配置**：每种制度的具体参数（比例、层数、封顶等）通过 `calc_rule_json` 配置驱动
- **可追溯**：每笔奖金必须记录计算明细，可回溯到具体订单
- **可回滚**：发生退货时，需要精确回退相关奖金
- **高性能**：大型团队（数万节点）的奖金计算需要异步+分批处理

```
推荐架构：策略模式 + 异步消息队列 + 定时任务日结
```

### 6.2 团队树查询优化（十万+ 规模）

面对十万级以上会员，树结构查询必须做针对性优化：

**存储方案：路径枚举 + Redis 缓存**

```
ancestor_path 字段：如 "1,5,23,87,156"
查询所有下线：SELECT * FROM ds_member WHERE FIND_IN_SET(87, ancestor_path)
```

**十万+ 规模专项优化：**

| 优化点               | 方案                                                                                                       |
| -------------------- | ---------------------------------------------------------------------------------------------------------- |
| **树结构缓存** | 团队树全量加载到 Redis（Sorted Set / Hash），避免每次查询遍历 MySQL                                        |
| **路径索引**   | MySQL 中`ancestor_path` 建立前缀索引，或用辅助表 `ds_member_path` 配合 `IN` 查询替代 `FIND_IN_SET` |
| **分页加载**   | 前端团队树采用懒加载，按层级展开，每次只加载当前层子节点                                                   |
| **业绩汇总**   | 团队业绩采用「预计算」模式，每日定时汇总，避免实时递归聚合                                                 |
| **分表策略**   | 会员主表按月分表或按会员 ID 取模分表，配合 ShardingSphere                                                  |
| **奖金计算**   | 异步批量处理，按团队分片并行计算，通过 RabbitMQ 削峰                                                       |

### 6.3 数据库额外优化

- MySQL 8.0 利用 CTE（`WITH RECURSIVE`）做递归查询，比传统自连接更高效
- 会员表关键字段（`phone`, `member_no`, `recommend_id`）建立联合索引
- 热点数据（会员等级、余额）用 Redis 缓存，TTL 按业务设定
- 写操作走数据库，读操作优先走 Redis，采用 Cache Aside 模式
